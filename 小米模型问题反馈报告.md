# mimo-v2.5-pro 英文金融文本结构化输出异常报告

## 反馈对象：小米大模型团队
## 报告日期：2026-05-11

---

## 一、实验背景

我们是一个 AI 驱动的金融市场状态分析系统（fin_robot）。系统的核心管线流程为：

1. 从 RSS 源（BBC、CNBC、WSJ、36氪等）采集英文和中文新闻
2. 调用大模型对文本进行金融相关性过滤（输出 JSON）
3. 调用大模型对金融相关文本进行信号提取（输出结构化 JSON，含分类、情感、重要性、置信度、摘要、逻辑推理、风险点等 15 个字段）
4. 将结构化信号写入数据库，供量化分析引擎使用

默认使用的模型是小米 `mimo-v2.5-pro`。在测试中发现该模型在**英文金融文本的结构化 JSON 输出**场景下存在异常。

---

## 二、问题描述

### 2.1 现象

在对小米模型进行批量 RSS 新闻处理时（7 条新闻，limit_per_feed=1），观察到：

- 中文金融新闻：信号提取正常，返回完整闭合的 JSON
- 英文金融新闻（BBC、CNBC、WSJ）：全部触发超时熔断（45 秒 timeout），模型未能返回有效 JSON

### 2.2 初步排查

随后进行了**单条英文金融文本的独立测试**（绕过 Fallback 逻辑，直接调用 OpenAI 兼容接口）。

**测试文本：**


- 34.2 秒的耗时异常（预期应在 3-10 秒内完成）

## 五、结论分析

### 5.1 问题定位

小米模型 `mimo-v2.5-pro` 在英文金融文本的结构化 JSON 输出场景下存在**输出中途中断**的问题。具体表现为：

- 对于中文新闻（较短文本），可以完整输出
- 对于英文新闻（较长文本），输出在生成长文本字段（如 summary/logic）时突然停止，导致 JSON 不完整
- 34.2 秒的耗时远超正常推理时间（对比 DeepSeek 4.6 秒），表明模型在生成过程中出现了 stall

### 5.2 已排除的因素

| 因素 | 是否问题根源 | 证据 |
|------|------------|------|
| Prompt 设计缺陷 | 否 | DeepSeek 和 OpenRouter 使用相同 Prompt 均能正确执行 |
| 字段缩写过于复杂 | 否 | 小米模型正确输出了所有缩写字段名 |
| 语言解绑规则混淆模型 | 否 | 小米模型正确混合使用英文（sum）和中文（枚举值） |
| max_tokens 限制（2000） | 否 | 实际输出仅 526 tokens，远未达到上限 |
| 网络/API 超时 | 否 | 请求成功返回 HTTP 200 |

### 5.3 推测原因

模型在生成长序列文本（特别是包含英文金融术语的段落）时，注意力机制或解码器层出现了不稳定，导致提前终止生成。这可能是针对英文金融领域的长文本生成能力优化不足所致。

## 八、生产环境影响评估：完整 RSS 流程对比

### 8.3 语言解绑的实际效果验证

从 DeepSeek 入库的 84 条信号中可以清晰看到语言解绑规则生效：

- 英文新闻的 `sum` 和 `log` 字段保持英文原语输出，未翻译
- 中文新闻的对应字段保持中文
- 枚举值字段（emo/r_lvl/bias 等）中英文场景均使用中文
- 缩写字段（cat/emo/imp/conf/hor/mkts/assets/sects/themes/sum/log/risks/r_lvl/bias）被正确理解和填充

### 8.4 优化收益总结

针对小米模型输出的问题，我们实施了两项优化措施：

**优化 1：字段缩写（全拼 → 缩写）**
- 目标：减少模型输出 token 量，降低截断概率
- 实际收益：每响应节省约 99 chars（20.5%），89 次调用合计节省约 5,000 tokens
- 对 DeepSeek 的边际收益：费用节省极小（deepseek-chat ¥1/百万 token），但响应速度略有提升

**优化 2：语言解绑规则**
- 目标：避免模型将英文金融文本翻译为中文再输出，减少 token 消耗和信息损失
- 实际收益：节省约 60-150 chars/调用的翻译输出，且保持了金融术语的原始精度
- 对 DeepSeek 的附加价值：规则被完整执行，中英文混合输出稳定

### 8.5 核心结论

小米模型 mimo-v2.5-pro 在英文金融文本的结构化 JSON 提取场景下存在严重的输出中断问题，具体表现为：

1. **输出中途停止**：在生成长文本字段（如 logic/summary）时突然终止，JSON 不闭合
2. **处理耗时异常**：英文单条请求耗时 34 秒，远超正常模型的 3-5 秒
3. **英文全覆盖失败**：在生产环境中，所有英文源（BBC、CNBC、WSJ、MarketWatch）的信号提取均失败

上述问题在相同的 Prompt、相同的并发管线、相同的 temperature/max_tokens 配置下，使用 DeepSeek deepseek-chat 模型均可正常完成（4.6 秒返回完整闭合 JSON），排除了 Prompt 设计和系统架构因素。

**对小米团队的建议**：建议排查 mimo-v2.5-pro 在英文金融长文本场景下的解码稳定性，特别是长序列生成时的注意力机制和提前终止条件设置。
